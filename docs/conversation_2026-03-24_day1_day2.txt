## Actually, Yes -- Docker Solves This Cleanly

The Xavier runs Ubuntu 20.04 (JetPack 5.x). You can't change the **host OS** to 22.04, but you absolutely can run ROS2 Humble inside a **Docker container** with full GPU access.

NVIDIA explicitly supports this. They even maintain pre-built containers for exactly this use case:

| Approach | Works? | Details |
|---|---|---|
| Native Ubuntu 22.04 install | No | JetPack 6 = Orin only, Xavier stuck at 20.04 |
| Compile ROS2 Humble from source on 20.04 | Technically yes | Pain, dependency hell, not recommended |
| **Docker container (Ubuntu 22.04 + Humble)** | **Yes, fully supported** | NVIDIA `jetson-containers` project, GPU passthrough works via `--runtime nvidia` |
| RoboStack (conda) | Possible | Distro-agnostic ROS2 via conda, but less mature |

### How Docker on Xavier Works

```
┌─────────────────────────────────────────────┐
│ Jetson AGX Xavier                            │
│ Host: Ubuntu 20.04 + JetPack 5.x            │
│                                              │
│  ┌─────────────────────────────────────┐     │
│  │ Docker Container                    │     │
│  │ Ubuntu 22.04                        │     │
│  │ ROS2 Humble                         │     │
│  │ MoveIt2, YOLO11, our orchestrator   │     │
│  │ CUDA 11.4 (passthrough from host)   │     │
│  │ 512 CUDA cores + 64GB unified mem   │     │
│  └─────────────────────────────────────┘     │
│                                              │
│  ROBOCOB's ROS1 stack can run separately     │
│  on host or in its own container             │
└─────────────────────────────────────────────┘
```

NVIDIA's `jetson-containers` project (`github.com/dusty-nv/jetson-containers`) has pre-built images like `dustynv/ros:humble-desktop-l4t-r35.x.x` that are tested on Xavier with JetPack 5.

The `--runtime nvidia` flag gives the container direct access to the GPU. YOLO11s inference on Xavier's 512 Volta CUDA cores would run comfortably.

### So We Actually Have Two Viable Deployment Options

**Option A: Your PC runs everything (Path C as planned)**
- Simplest, no Docker complexity
- Laptop connects to Doosan via Ethernet, Hand-E via USB, camera via USB
- YOLO runs on laptop GPU or CPU
- Xavier untouched

**Option B: Xavier runs our stack via Docker**
- More powerful GPU for inference (Volta 512 cores + 64GB unified memory)
- No need to carry a laptop to the lab
- Xavier handles compute, talks to Doosan controller on same subnet (192.168.3.x -- already on the same network!)
- Camera can stay USB-connected to Xavier (it already has USB ports)
- Hand-E RS485 also connects to Xavier's USB

**Option B is actually cleaner for the final demo** -- everything runs on one box that's already physically co-located with the robot and on the same network.

### Bottom Line

I was wrong to say we can't use the Xavier. Docker makes it fully viable, and it's NVIDIA's recommended approach for running newer software on older JetPack versions. This gives us flexibility -- develop on your laptop (WSL2), deploy to Xavier (Docker) for the real demo.

- **Doosan + Robotiq ROS2 driver'ları var** -- `doosan-robot2` ve `ros2_robotiq_gripper` paketleri mevcut

### Dikkat edilmesi gerekenler

**1. Hand-E iletişim yolu belirsiz**
ROBOCOB'da Hand-E, Doosan'ın tool flange'ından RS485 ile bağlı. Bu sinyal kontrolcüden mi çıkıyor, yoksa doğrudan mı? Lab'da trace etmek lazım. Eğer kontrolcü üzerinden geçiyorsa, Doosan API'si üzerinden erişmek gerekebilir (direkt RS485 olmayabilir).

**2. Doosan controller modu**
Teach pendant'ta **"remote mode"** aktif olmalı ki dışarıdan komut kabul etsin. Bunu lab'da ilk kontrol edecek şeylerden biri yapmalıyız. Aksi halde hiçbir komut çalışmaz.

**3. USB passthrough Docker'da test edilmeli**
Kamera (USB) Docker'dan `--privileged` ile erişilebilir ama mutlaka lab'da test edilmeli. Bazen USB3 bandwidth sorunları oluyor.

**4. "Sim'de çalışan = reelde çalışır" kısmen doğru**
- Joint trajectory'ler: evet, direkt çalışır
- Gripper kuvvet feedback'i: sim'de yok, reelde farklı davranacak
- Kamera kalibrasyonu: sim'deki ideal kamera vs gerçek lens distortion
- Network latency: Docker → Doosan controller arasında sim'de olmayan gecikme olabilir

**5. Kamera aparatı kritik**
Vibrasyon minimize edilmeli, yoksa hareket sırasında YOLO detection bozulur. Aparat tasarımında sertlik önemli.
